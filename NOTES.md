## 1. Cloudflare блокирует requests — почему?

**Ассоциация:** ты подходишь к клубу в пижаме. Охранник (Cloudflare) смотрит на тебя и говорит "не-а". Потому что `requests` шлёт заголовок `User-Agent: python-requests/2.32.5` — это как прийти в пижаме с надписью "Я БОТ". Cloudflare видит → 403.

А reqres.in ещё и ключ требует (`x-api-key`). Без ключа даже в костюме не пустят.

**Решение:** ключ в `.env`, подставляем в заголовки через конфиг. Если API поменяет правила — правим конфиг, не код.

**Ошибки новичков:**
- Хардкодить ключ в коде → попадёт в git → утечка
- Думать что 403 = "сервер сломался" → на самом деле это защита, читай тело ответа

## 2. Зачем `__init__.py`?

**Ассоциация:** табличка "ОТКРЫТО" на двери магазина. Без неё Python проходит мимо папки и не заглядывает внутрь. Есть табличка → заходит, можно импортировать.

```python
from tests.api.checks.user_checks import UserChecks
#     ^^^^  ^^^  ^^^^^^ — в каждой из этих папок должен быть __init__.py
```

У нас файлы пустые — просто разрешают импорты. Можно писать в них реэкспорт, но пока не нужно.

**Ошибки новичков:**
- Забыл создать → `ImportError` и непонятно почему — Python просто не видит папку
- Создал в `tests/` но не в `tests/api/checks/` → вложенный импорт сломан
- Написал код в `__init__.py` когда не нужно → сложнее дебажить

## 3. Параметризация в pytest

**Ассоциация:** связка ключей. У тебя один замок (тест) и 3 ключа (данные). Вместо того чтобы писать 3 одинаковых теста — берёшь связку и пробуешь каждый по очереди. Один тест, три прогона.

```python
@pytest.mark.parametrize("id", [1, 2, 3])
def test_resource(id):
    response = api.get(f"/api/unknown/{id}")
    assert response.status_code == 200
# pytest создаст: test_resource[1], test_resource[2], test_resource[3]
```

**Ошибки новичков:**
- `parametrize` а не `parameterize` (без второго "e"!) → декоратор молча не работает
- Имя в строке не совпадает с аргументом: `@pytest.mark.parametrize("user_id", ...)` но `def test(id):` → тест не запустится
- Копипастить 10 одинаковых тестов вместо параметризации → на ревью скажут "зачем?"

## 4. Что такое `self`?

**Ассоциация:** слово "я" в разговоре. Когда ты говоришь "я открою дверь" — "я" указывает на тебя. Когда объект вызывает свой метод — `self` это его "я".

```python
rex = Dog("Рекс")
rex.bark()
# Python под капотом делает: Dog.bark(rex)
#                                     ^^^ этот rex и становится self
# внутри метода: self.name → "Рекс"
```

**Аналог во Flutter:** `this`. Но в Dart можно не писать, а в Python `self` — обязательно в каждом методе, иначе ошибка.

**Ошибки новичков:**
- Забыл `self` → `TypeError: bark() takes 0 positional arguments but 1 was given` — самая частая ошибка новичков в Python
- Написал `name = "Рекс"` вместо `self.name = "Рекс"` → создал локальную переменную, которая умрёт после `__init__`, а объект останется без имени
- Назвал `this` вместо `self` → работает, но любой питонист при ревью скажет "переделай"

## 5. Почему константы БОЛЬШИМИ БУКВАМИ?

**Ассоциация:** табличка "НЕ КУРИТЬ" на стене. Написана капсом, потому что это правило, которое не меняется. Так же `VALID_EMAIL` — задали раз, больше не трогаем.

```python
VALID_EMAIL = "eve.holt@reqres.in"    # константа — КАПС, не трогай
user_name = input("Имя?")             # переменная — маленькие, меняется
```

`from tests.api.test_data import VALID_EMAIL` = "из файла test_data возьми конкретную штуку, а не весь файл целиком".

**Ошибки новичков:**
- `validEmail` (camelCase) → это Java/Dart стиль, в Python пишут `VALID_EMAIL` через подчёркивания
- `valid_email` маленькими → кто-то подумает что обычная переменная и перезапишет
- Python **не запрещает** менять константу — это только договорённость между людьми, язык не защитит

## 6. Разбор теста: test_create_user

```python
response = api_client.post("/api/users", json={...})  # запрос → ответ в переменную
user_checks.check_user_created(response, ...)          # проверка ответа
```

**Ассоциация:** ты отправляешь письмо (post) и получаешь ответ в конверте (response). Потом отдаёшь конверт проверяющему (user_checks), и он смотрит: "статус правильный? имя совпадает? id не пустой?"

`api_client` склеивает `base_url + path` → `https://reqres.in/api/users` и шлёт запрос. В `response` лежит всё: статус-код, заголовки, тело (JSON).

## 7. Почему IDE ведёт в conftest, а не в UserChecks?

**Ассоциация:** ты заказал пиццу. Хочешь узнать "из чего пицца?" — но трекер показывает **курьера**, а не повара. Потому что пицца пришла к тебе через курьера.

```python
def test_create_user(self, api_client, user_checks):
#                                      ^^^^^^^^^^^
# IDE: "откуда user_checks?" → из conftest.py (курьер)
```

```python
# conftest.py — "курьер"
@pytest.fixture
def user_checks():
    return UserChecks()   # ← а вот тут реальный объект (повар)
```

```python
# checks/user_checks.py — "повар"
class UserChecks:
    def check_user_created(self, ...):  # ← метод живёт тут
```

**Цепочка кликов в IDE:** `user_checks` в тесте → conftest (фикстура) → кликни на `UserChecks()` → попадёшь в класс с методами. Два клика вместо одного.

**Аналог во Flutter:** кликаешь на переменную из GetIt → IDE ведёт к `registerSingleton`, а не к самому классу. Та же логика.

## 8. `assert value` — проверка на truthiness

**Ассоциация:** охранник на входе. `assert user.id` = "покажи документ". Если документ есть (непустая строка, число больше нуля) — проходи. Если нет документа (пустая строка, None, ноль) — стой.

```python
assert "hello"   # OK — есть что показать
assert ""        # FAIL — пусто
assert 0         # FAIL — ноль считается "пустым"
assert None      # FAIL — ничего нет
assert 42        # OK
```

Аналог `isNotEmpty` из Flutter, но универсальнее — работает с любым типом, не только строками.

**Ошибки новичков:**
- `assert user.id == True` → строка `"123"` это не `True`! Тест упадёт, хотя id есть
- `assert user.id is not None` → пропустит пустую строку `""`, а она тоже "пустая"
- Забыть сообщение: `assert user.id` → при падении видишь только "AssertionError" без пояснения. Лучше: `assert user.id, "id пустой"`

## 9. Почему Python для тестов, а не TS или Java?

**Ассоциация:** швейцарский нож — не самый быстрый, но подходит под всё.

**Главное:** один язык для API (requests) + UI (Playwright) + нагрузка (Locust). В TS пришлось бы миксовать подходы.

- pytest — лучший тест-фреймворк (фикстуры, параметризация, плагины). Jest/Mocha слабее
- Вакансий Python QA в 4-5 раз больше чем TS QA на hh.ru
- Python лаконичнее: `response = requests.get(url)` vs `const response = await fetch(url)`
- Переход из Flutter/Dart → Python логичнее чем → TS (не нужно учить JS-экосистему)

**Когда TS лучше:** вся команда на TS, component testing React/Vue.

**Ошибки новичков:**
- "Java, там больше денег" → Java QA вымирает, порог входа высокий, фреймворки тяжёлые
- "Python медленный" → для тестов не важно, bottleneck — сеть и браузер, не язык

## 10. Зачем Docker для тестов?

**Ассоциация:** коробка для LEGO. Без неё модель развалится по дороге. Docker — упаковка: Python, Playwright, браузер — всё внутри, работает одинаково у всех.

- Коллега запускает `docker-compose up` — не нужно ставить Python, Playwright, браузер
- CI/CD запускает тесты в контейнере — гарантия одинакового окружения
- 80% вакансий Python QA 300К+ требуют Docker — строчка в резюме

**Когда не нужен:** ты один, тесты только на твоей машине, никому не показываешь.

**Ошибки новичков:**
- Думать что Docker ускоряет тесты → нет, он про воспроизводимость, не про скорость
- Забыть `.dockerignore` → `.git/` попадает в образ, сборка 10 минут
